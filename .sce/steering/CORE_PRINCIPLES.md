# 核心开发原则（基准规则）

> 这里只保留跨周期、跨 Spec 仍成立的规则。短期状态写入 `CURRENT_CONTEXT.md`，详细制度与示例写入 `docs/steering-governance.md`。

## 1. Steering 分层必须稳定

- `CORE_PRINCIPLES.md` 只放长期原则，不放任务、历史、版本脚注、发布记录。
- `ENVIRONMENT.md` 只放项目运行环境、目录约定、发布触发方式等项目级规则。
- `CURRENT_CONTEXT.md` 只放当前阶段最小必要上下文，不保留长流水。
- `RULES_GUIDE.md` 只放职责边界与迁移规则，不承载详细流程。

## 2. 所有需求先落 Spec

- 先建 Spec，再写 `requirements.md`、`design.md`、`tasks.md`，再进入实现与验证。
- 脚本、报告、诊断、验证产物都归档到对应 `.sce/specs/<spec>/`，不要散落根目录。

## 3. 默认自主闭环推进

- 默认执行链路是：分析 -> 修改 -> 测试 -> 修复 -> 文档同步 -> 交付。
- 非阻断性问题不得反复等待确认；只有外部权限、不可恢复错误、高风险生产影响或目标冲突时才升级给用户。

## 4. 复用已有 SCE 能力，不得平行造轮子

- steering 只能引用现有机制，不能在规则层再发明一套并行流程。
- 缺陷沉淀、临时兜底、发布阻断统一走 `errorbook` 能力，不再额外定义平行的缺陷学习或发布治理机制。
- 若接管项目中已存在自定义缺陷复盘册、故障账本或问题经验库，必须统一盘点并吸收到 `.sce/errorbook`，禁止长期并存混用。
- 会话、Scene、release gate、problem closure 等治理能力，应直接复用现有脚本、门禁和工作流。

## 5. 质量问题必须追根，不允许伪修复

- 禁止通过跳过测试、关闭校验、吞错回退、降级关键路径来伪装修复成功。
- 两轮修复仍未通过验证时，必须切换到调试与证据模式，先补足可观测信号再继续改。
- 临时兜底只能止血，必须同时记录退出条件、清理任务和截止时间。

## 6. 测试、文档、代码必须同步闭环

- 代码变更必须跑相关验证；发布前不得忽略失败。
- 重要功能、命令、配置变化必须同步更新 README、用户文档或发布说明。

## 7. 单一事实源优先

- 每类关键数据只保留一个权威来源，避免双写、镜像字段和并行配置。
- 前后端不一致时，优先对齐权威模型，而不是在非权威侧追加补丁字段。

## 8. Git 托管与发版门禁默认启用

- 若仓库已配置 GitHub/GitLab 远端，交付和发布前默认要求工作区干净、分支有 upstream、与远端同步。
- 正式发布遵循 tag 驱动；相关门禁统一通过现有 release workflow 与脚本执行。

## 9. 开发测试环境默认授权

- 在开发和测试环境，AI 默认可读写项目、运行构建测试、访问必要网络资源并完成诊断。
- 仅在生产风险、敏感数据或不可逆外部影响场景下提高确认等级。

## 10. Steering 自身必须定期净化

- 每周、每次发布前、每次大版本/大 Spec 收尾后，都要运行 `npm run audit:steering`。
- 审计目标不是保留更多规则，而是持续做四件事：合并重复、迁移错层、归档短期内容、删除失效条目。
- steering 若无法在最短时间内说明“什么长期有效、什么当前有效、什么应迁走”，就说明它已经开始腐化，必须重写。

## 11. 业务场景未知时必须先澄清，禁止直接彻底禁用

- 不了解业务场景、模块、页面、实体或业务约束时，默认动作是先补上下文、缩小范围、生成澄清问题。
- 禁止把“未知业务场景”直接等同于 `deny`、`disable`、answer-only 或其他一刀切兜底禁用。
- 只有在明确命中安全/权限/合规/破坏性规则后，才允许进入阻断；否则必须先完成业务范围澄清。
- 这条规则适用于所有接入 SCE 的项目、模式和交互面，不允许按项目例外绕过。

## 12. 禁止盲改问题，必须先建立问题契约和证据

- 修改问题前，必须先明确现象、复现条件、影响范围、预期行为和验证方式。
- 缺少日志、数据、接口样本、最小复现或问题契约时，应先补证据，不得靠猜测连续改代码碰运气。
- 若两轮修改仍未收敛，必须回到调试、定位和根因分析，禁止在未理解问题前盲目扩大改动面。

## 13. Steering 条目变更必须先评估，禁止随意增删

- 新增、删除或重写 steering 条目前，必须先评估它是否真属于长期原则，是否与现有条目重复，是否应迁移到 `CURRENT_CONTEXT.md`、Spec 或项目文档。
- steering 变更必须说明触发原因、适用范围以及与现有规则的关系；未经评估，不得把临时偏好、短期任务或偶发结论直接固化进去。
- 接管、升级和治理脚本只能补齐基线、修复漂移，不能把未经评估的项目习惯直接塞进 steering。

## 14. 问题修复时前后端接口不一致默认以后端契约为准

- 在修改问题的场景下，若前端调用后端 API 出现路径、方法、字段、状态码或响应结构不匹配，默认以后端现有接口契约为准。
- 除非明确要求新建接口或修改后端接口，否则禁止为了迁就前端错误调用去随意改后端实现或契约。
- 默认优先修正前端请求、映射、类型和兼容处理，使其与后端接口保持一致；若怀疑后端契约错误，应先确认再改。

## 15. 单文件规模过大必须触发重构评估，禁止无限堆积

- SCE 应为每个项目定期评估代码规模分布，并给出项目级的重构参考节点；禁止假设所有项目都适用同一个固定行数阈值。
- 若项目尚未建立自己的阈值，默认参考源文件 `2000 / 4000 / 10000` 行三档触发：分别对应“必须评估”“必须发起重构收敛”“进入红线区”。
- 达到项目级或默认阈值后，后续改动必须优先评估拆分模块、服务、命令面或数据职责；超过重构/红线阈值时，不得继续无计划堆积复杂度。
- 项目开始较小时，阈值应更早触发；项目进入长期演进后，也必须按周或发布前重新评估，而不是让早期设定永久失效。
- 行数阈值只是强触发信号，不代表低于阈值就可以忽略耦合、职责混杂、测试失控和理解成本问题；若复杂度已明显失控，应提前启动重构。
